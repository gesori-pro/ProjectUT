💡 이 모든 것은 명확한 요청 하나로 시작되었습니다

❌ 나쁜 요청(기획) 예시

"선택지 위치 좀 수정해줘 (안 예뻐)"

개발자가 겪는 어려움:

  • ❓어떻게 수정할지 불명확 (수치 부재)
  • ❓현재 문제가 무엇인지 모름 (맥락 부재)
  • ❓원하는 결과가 모호함 (기준 부재)

✅ 좋은 요청(기획) 예시

"3인 상태에서 1번 선택지가 아주 조금만 더 오른쪽으로 가면 될 거 같아 한 5px정도"

개발자가 좋아하는 이유:

  • ✓구체적인 상황 설명 → 3인 상태 = 조건
  • ✓정확한 대상 지정 → 1번 선택지 = 대상
  • ✓명확한 수치 제공 → 5px = 값

구글 엔지니어링 리더의 AI 코딩 원칙

🗣️ Addy Osmani (Chrome Engineering Lead)

  • 🏁 코드보다 계획 먼저
    바로 코딩하지 말고 spec.md로 요구사항 정리
  • 🔄 작은 단위 반복
    한 번에 완성하려 하지 말고 단계별로 구현
  • 🗺️ 충분한 맥락 제공
    제약조건, 피해야 할 패턴을 명시적으로 전달
  • 👨‍🏫 Human-in-the-loop
    AI 코드는 주니어 코드라 생각하고 직접 리뷰
  • 💾 자주 커밋
    언제든 돌아갈 수 있게 롤백 포인트 확보

실제 소통 과정 #1

❌ 느낌적인 요청

"좌표계가 이상한듯 한데"

→ 개발자가 코드를 처음부터 다시 봐야 함

✓ 현상 기반 요청

"두 캐릭터인데, 왼쪽 캐릭터 선택지가 너무 왼쪽에 쏠려있어"

→ 문제 재현 경로와 현상이 명확함

실제 소통 과정 #2

❌ 모호한 요청

"선택지가 캐릭터 위치에 맞게"

→ "맞게"의 기준이 기획자와 개발자가 다름

✓ 로직 기반 요청

"X축은 유지해 주고 Y축만 변경해줘"

→ 수정해야 할 로직의 범위를 정확히 지정

개발은 '대화'하며 완성됩니다

실제 위치 조정 과정 (핑퐁)

  • 1 "조금 오른쪽으로" → 개발자: +25px 이동
  • 2 "5px 정도 더" → 개발자: +5px 이동
  • 3 "한 번만 더 5px" → 개발자: +5px 이동
  • ✓ "딱 좋아!" → 컨펌 완료 🎉

💡 한 번에 완벽할 순 없습니다. 빠른 피드백과 수정이 핵심!

버그 리포팅의 정석

  • ❌ "안 돼요 / 버그 있어요"
  • ✓ "버튼을 클릭해도 [진행] 이벤트가 발생하지 않아요"
  • ✓ "3개 중 1번 선택지만 안 보여요"
  • ✓ "배경 전환 효과 때문에 UI가 가려져요"

증상 + 재현 방법을 말해주면 개발자가 춤을 춥니다 💃

큰 기능은 잘게 쪼개서 요청하기

"대화 순환 시스템" (너무 큼) 👉 쪼개기

  • 1 "한 번 선택한 선택지는 다시 안 나오게" (once flag)
  • 2 "남은 캐릭터 수에 따라 레이아웃 변경" (if 문)
  • 3 "선택지가 다 떨어지면 다음 씬으로" (fallback)

💡 개발자가 로직을 짜기 쉽게 논리 단계를 나눠주세요!

여러분의 명확한 기획이 멋진 기능이 됩니다!

프롬프트 작성 = 기획서 작성

💬 AI 프롬프트

  • 📍"3인 모드일 때" (조건)
  • 🎯"1번 선택지를" (대상)
  • 📐"5px 오른쪽으로" (값)

📝 프로그래머 기획서

  • 📍[조건] 3인 레이아웃 활성화 시
  • 🎯[대상] Choice_Btn_01 객체
  • 📐[동작] X position += 5

AI와 대화하는 연습은 곧 개발자와 일하는 근육을 키우는 과정입니다

핵심 정리

  • 1️⃣ 두루뭉술한 느낌 ❌ → 명확한 수치/상태 ✓
  • 2️⃣ "그냥 안 돼요" ❌ → "어떻게 하니 안 돼요" ✓
  • 3️⃣ 한 방에 다 ❌ → 하나씩 차근차근 ✓
  • 4️⃣ 명령 ❌ → 협업과 조율 ✓

좋은 기획자는 '말이 통하는' 기획자입니다.
